# Testcase Generator

生成适用于真实项目交付的测试设计产物和结构化测试用例，强调可追溯、可执行、可验证，而不是单纯堆砌数量。

## 这个 Skill 做什么

本 Skill 用于把原始输入逐步转化为适合测试设计与评审的产物。

支持的输入包括：
- 需求文档
- PRD
- 用户故事
- 功能描述
- API 规范
- 源代码
- 缺陷说明或回归背景
- 本地 Markdown / PDF 需求文件

支持的结果包括：
- 结构化测试用例
- 测试场景清单
- 测试点与边界条件清单
- 需求到用例的追溯矩阵
- 面向状态流转的测试设计结果
- 面向评审的质量检查与缺口说明

## 核心工作流

当任务需要完整分析深度时，按以下顺序执行：

1. 识别输入类型并进行规范化。
2. 提取可测试需求、约束和关键业务规则。
3. 当提供代码或 API 时，补充分析实现逻辑、异常处理和契约风险。
4. 当系统存在明显状态流转或复杂业务规则时，建立领域模型与状态模型。
5. 当复杂度足够高时，进一步执行 MBT 导向测试设计。
6. 生成结构化测试用例、场景清单和追溯输出。
7. 在交付前依据质量检查项进行校验。

如果用户只要轻量结果，可以走降级路径，例如：
- 快速测试点清单
- 场景梳理
- 边界条件清单
- 回归建议

## 执行模式

根据输入与任务深度选择合适模式。

### 1. 需求驱动模式
适用于用户提供：
- PRD
- 需求文档
- 用户故事
- 自然语言功能描述

重点关注：
- 功能拆解
- 验收条件
- 边界条件
- 正常与异常场景

### 2. 代码辅助模式
适用于用户提供：
- 源代码
- 仓库文件
- 接口实现
- 服务逻辑
- API 处理代码

重点关注：
- 控制流
- 数据流
- 异常路径
- 契约风险
- 隐藏分支与潜在缺陷点

### 3. MBT 导向模式
适用于系统存在以下特征：
- 复杂状态流转
- 审批链路
- 订单或支付生命周期
- 角色权限分支
- 规则密集型业务流程

重点关注：
- 业务实体与状态模型
- 转换条件与非法路径
- 覆盖准则
- 在保证覆盖的前提下控制用例规模

### 4. 回归聚焦模式
适用于用户提供：
- 缺陷描述
- 修复背景
- 补丁上下文
- 事故复盘信息

重点关注：
- 受影响路径
- 邻近风险
- 边界回归
- 负向场景
- 接口兼容性

## 执行规则

执行时始终遵守以下规则：

- 不要凭空编造需求、业务规则或预期结果；如果必须推断，明确标记为“推断”或“假设”。
- 始终保留来源与结论之间的追溯关系。
- 优先写出可执行、可验证的预期结果，避免空泛描述。
- 明确区分“已确认信息”和“推断信息”。
- 当关键信息缺失时，先报告缺口，再决定是否输出降级版本。
- 如果代码实现与需求描述冲突，优先把代码视为当前实现，同时显式指出需求不一致。
- 质量优先于数量，避免生成大量低价值或重复用例。
- 术语一旦确定，全程保持一致。

## 最小交付协议

当任务未展开为完整模板实例时，至少必须输出以下字段：

- 模块或功能名称
- 场景标题
- 前置条件
- 测试数据
- 执行步骤
- 预期结果
- 优先级
- 用例类型
- 追溯引用
- 假设项、歧义点或待确认项

如果缺少以上关键字段中的 3 项及以上，不得将结果声称为“正式测试用例”，只能标记为“测试草稿”或“测试点清单”。

## 质量评审必做步骤

在交付结构化测试用例前，至少完成以下评审动作：

1. 检查高风险需求是否被优先覆盖。
2. 检查每条用例是否具备可执行步骤与可验证预期结果。
3. 检查是否存在明显重复或仅换表述的冗余用例。
4. 检查边界、负向、权限、状态流转场景是否遗漏。
5. 检查所有推断内容是否被明确标记。
6. 检查是否存在输入缺陷导致的不可判定项，并在输出中显式保留。

## 输出要求

在生成完整结构化测试结果时，尽量包含以下字段：

- 模块或功能名称
- 场景标题
- 前置条件
- 测试数据
- 执行步骤
- 预期结果
- 优先级
- 用例类型
- 追溯引用
- 假设、歧义点或待确认项说明

常见用例类别包括：
- 正常路径
- 异常路径
- 边界值
- 等价类
- 权限/角色场景
- 状态流转场景
- 数据校验场景
- 接口契约场景
- 回归场景

## 质量门禁

在最终交付前，至少检查：

- 可测试需求是否得到覆盖
- 场景是否具备可执行性
- 预期结果是否可观察、可验证
- 边界条件是否被覆盖
- 负向路径是否被遗漏
- 重复用例是否被控制
- 是否建立了必要的追溯关系
- 假设项是否被清楚标记
- 术语与结构是否前后一致

如果输入质量不足以支撑正式交付，先返回：
1. 风险摘要
2. 缺失信息清单
3. 如仍有价值，再给出降级版测试草稿

## 工作流选择建议

可按以下方式判断：

- 仅提供需求文本 → 采用需求驱动模式
- 同时提供需求和代码 → 采用需求驱动 + 代码辅助模式
- 存在状态流转、生命周期或复杂规则 → 增加 MBT 导向模式
- 提供缺陷、补丁或事故背景 → 采用回归聚焦模式
- 用户只想快速获得思路 → 走轻量路径，而不是强行跑完整流水线

## 输出风格要求

执行本 Skill 时：

- 明确说明哪些内容已确认，哪些属于推断
- 保持输出结构化、便于评审
- 语言尽量简洁，但信息要完整
- 不写空泛套话
- 清晰暴露测试风险、歧义点与覆盖缺口

## 交付深度策略

根据用户需求匹配合适的输出深度。

### 轻量交付
适用于：
- 快速讨论
- 需求评审前期
- 只需测试点或补充建议

示例：
- 测试点清单
- 场景清单
- 边界条件列表
- 回归建议

### 标准交付
适用于：
- 常规模块测试
- Sprint 级测试设计
- API / 页面 / 功能评审

示例：
- 结构化测试用例
- 需求到测试映射
- 优先级分层场景
- 边界与异常覆盖清单

### 完整交付
适用于：
- 高风险业务流程
- 复杂状态流转系统
- 支付、审批、订单、结算、合规类场景
- 需要正式交付与追溯的测试设计任务

示例：
- 分阶段分析产物
- 领域/状态模型
- MBT 设计结果
- 完整测试用例集
- 追溯矩阵
- 质量总结

## 约束

- 本 Skill 通过结构化分析提高测试产出质量，但不能替代人工领域确认。
- 业务规则存在歧义时，应显式保留歧义，而不是悄悄猜测。
- MBT 深度应由复杂度驱动，而不是默认对所有小需求都启用。
- 输出规模应与业务风险和交付目标相匹配。
