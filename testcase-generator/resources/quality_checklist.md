# 质量检查指南

## 概述

本指南用于在每个阶段完成后进行主动质量检查，确保产物符合要求，避免幻觉内容进入后续阶段。

## 通用检查原则

1. **每阶段必检**：每个阶段完成后必须进行质量检查
2. **主动识别**：主动识别可能存在的幻觉或不符合要求的内容
3. **立即修正**：发现问题立即在当前阶段修正
4. **记录归档**：将检查结果记录到质量报告中

## 阶段一：需求预处理 - 质量检查

### 完整性检查
- [ ] 需求描述是否完整？有无遗漏的功能点？
- [ ] 用户角色是否识别完整？
- [ ] 业务流程是否覆盖所有场景？
- [ ] 验收标准是否齐全？

### 准确性检查
- [ ] 需求理解是否正确？有无曲解原意？
- [ ] 边界条件识别是否准确？
- [ ] 等价类划分是否合理？
- [ ] 业务规则描述是否准确？

### 可测试性检查
- [ ] 每个需求是否都可验证？
- [ ] 验收标准是否明确？
- [ ] 是否存在模糊的需求描述？

### 幻觉识别
- [ ] 是否存在未在原始需求中出现的新概念？
- [ ] 是否存在基于假设而非实际需求的内容？
- [ ] 是否存在超出需求范围的扩展？

## 阶段二：代码分析 - 质量检查

### 准确性检查
- [ ] 代码结构分析是否准确？
- [ ] 函数/方法签名是否正确？
- [ ] 数据流追踪是否准确？
- [ ] 控制流分析是否完整？

### 完整性检查
- [ ] 关键函数是否都已覆盖？
- [ ] 所有分支是否都已识别？
- [ ] 异常处理逻辑是否完整？

### 缺陷识别检查
- [ ] 缺陷点识别是否有依据？
- [ ] 严重性评估是否合理？
- [ ] 是否存在未经验证的推测性缺陷？

### 幻觉识别
- [ ] 是否存在对代码行为的错误理解？
- [ ] 是否存在未验证的实现细节描述？
- [ ] 是否存在基于猜测的缺陷推测？

## 阶段三：领域分析 - 质量检查

### 模型准确性
- [ ] 实体关系是否正确？
- [ ] 状态定义是否准确？
- [ ] 状态转换是否合理？

### 完整性检查
- [ ] 所有业务实体是否都已建模？
- [ ] 所有状态是否都已定义？
- [ ] 所有转换条件是否都已明确？

### 一致性检查
- [ ] 与需求阶段产物是否一致？
- [ ] 与代码分析阶段产物是否一致？
- [ ] 术语使用是否统一？

### 幻觉识别
- [ ] 是否存在未在需求中提到的实体？
- [ ] 是否存在未经验证的状态？
- [ ] 是否存在超出业务范围的关系？

## 阶段四：MBT设计 - 质量检查

### 模型正确性
- [ ] 状态机是否完整？
- [ ] 转换是否覆盖所有场景？
- [ ] 守卫条件是否准确？

### 覆盖准则合理性
- [ ] 覆盖级别是否合理？
- [ ] 测试用例数量是否适中？
- [ ] 是否存在路径爆炸风险？

### 可维护性
- [ ] 模型结构是否清晰？
- [ ] 是否易于扩展？
- [ ] 文档是否完整？

### 幻觉识别
- [ ] 是否存在无法执行的转换？
- [ ] 是否存在矛盾的转换条件？
- [ ] 是否存在无意义的覆盖要求？

## 阶段五：用例生成 - 质量检查

### 覆盖完整性
- [ ] 是否覆盖所有需求？
- [ ] 是否覆盖所有边界条件？
- [ ] 是否覆盖所有异常场景？
- [ ] 路径覆盖是否完整？

### 用例质量
- [ ] 用例是否清晰可执行？
- [ ] 输入数据是否明确？
- [ ] 预期结果是否确定？
- [ ] 执行步骤是否完整？

### 无冗余性
- [ ] 是否存在重复的用例？
- [ ] 是否可以合并相似用例？
- [ ] 优先级分配是否合理？

### 幻觉识别
- [ ] 是否存在无法执行的步骤？
- [ ] 是否存在无法验证的预期结果？
- [ ] 是否存在前后矛盾的用例？

## 质量检查流程

1. **自检**：执行者先自行检查
2. **交叉检**：如有条件，由他人交叉检查
3. **问题修复**：发现的问题在当前阶段修复
4. **确认签字**：检查通过后签字确认
5. **进入下一阶段**：获得批准后进入下一阶段

## 质量度量指标

| 指标 | 目标值 | 严重性阈值 |
|-----|-------|----------|
| 需求覆盖率 | 100% | < 95% |
| 边界覆盖率 | 100% | < 90% |
| 路径覆盖率 | > 80% | < 70% |
| 用例执行率 | 100% | < 95% |
| 缺陷检测率 | > 70% | < 60% |
