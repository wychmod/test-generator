# 测试用例格式参考

## 常用测试用例格式

### 格式一：简洁型
适用于快速执行的功能测试

```markdown
| 用例ID | TC-001 |
|--------|--------|
| 用例名称 | 验证用户登录成功 |
| 优先级 | P0 |
| 前置条件 | 用户已注册 |
| 测试步骤 | 1. 输入正确用户名<br>2. 输入正确密码<br>3. 点击登录 |
| 预期结果 | 登录成功，跳转首页 |
```

### 格式二：标准型
适用于详细的测试设计

```markdown
# TC-001: 验证用户登录成功

## 1. 基本信息
- **用例ID**: TC-001
- **测试类型**: 功能测试
- **优先级**: P0
- **模块**: 认证模块

## 2. 测试数据
| 字段 | 值 | 类型 |
|-----|---|-----|
| 用户名 | testuser | String |
| 密码 | Test@123 | String |

## 3. 测试步骤
1. 打开登录页面
2. 输入用户名: testuser
3. 输入密码: Test@123
4. 点击"登录"按钮

## 4. 预期结果
- [ ] 页面跳转至首页
- [ ] 显示用户头像
- [ ] 显示欢迎信息

## 5. 覆盖
- 需求: REQ-001
- 路径: P-001
```

### 格式三：Gherkin 格式
适用于 BDD 风格测试

```gherkin
Feature: 用户登录功能

  Scenario: 用户使用正确凭据登录
    Given 用户在登录页面
    When 用户输入用户名 "testuser" 和密码 "Test@123"
    And 点击登录按钮
    Then 登录成功，显示首页
    And 显示用户头像

  Scenario: 用户使用错误密码登录
    Given 用户在登录页面
    When 用户输入用户名 "testuser" 和密码 "wrongpass"
    And 点击登录按钮
    Then 登录失败，显示错误提示
    And 停留在登录页面
```

### 格式四：API 测试格式
适用于 API 测试

```markdown
## API 测试用例: POST /api/login

### 基本信息
- **用例ID**: TC-API-001
- **请求方法**: POST
- **Content-Type**: application/json

### 请求体
```json
{
  "username": "testuser",
  "password": "Test@123"
}
```

### 成功响应预期
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
    "userId": 12345,
    "expiresIn": 3600
  }
}
```

### 测试场景
| 场景 | username | password | 预期code | 预期message |
|-----|----------|----------|---------|------------|
| 正常登录 | testuser | Test@123 | 200 | success |
| 用户名不存在 |nouser | Test@123 | 401 | invalid credentials |
| 密码错误 | testuser | wrongpass | 401 | invalid credentials |
| 用户名为空 | (空) | Test@123 | 400 | username required |
```

### 格式五：数据驱动格式
适用于大量相似测试用例

```markdown
## 数据驱动测试: 用户年龄验证

### 测试用例模板
| 字段 | 值 | 说明 |
|-----|---|-----|
| 用例ID | TC-DD-### | 数据驱动格式 |
| 输入 | {age} | 参数化 |
| 预期 | {expected} | 参数化 |

### 测试数据
| age | expected | 场景 |
|-----|----------|-----|
| 0 | 失败 | 边界：最小值-1 |
| 1 | 失败 | 边界：最小值 |
| 17 | 失败 | 边界：未成年 |
| 18 | 成功 | 边界：成年 |
| 65 | 成功 | 正常值 |
| 120 | 失败 | 边界：超过最大 |
| 121 | 失败 | 边界：最大值+1 |
| -1 | 失败 | 异常：负数 |
| abc | 失败 | 异常：非数字 |
```

### 格式六：场景测试格式
适用于端到端场景测试

```markdown
## E2E 场景测试: 用户购物流程

### 场景概述
验证用户从浏览商品到完成订单的完整流程

### 用户故事
作为用户，我希望能够便捷地购买商品，以便快速完成购物

### 测试链路
```
浏览商品 → 加入购物车 → 确认订单 → 支付 → 支付成功
```

### 涉及页面/接口
1. 商品列表页
2. 商品详情页
3. 购物车页
4. 订单确认页
5. 支付页
6. 支付结果页

### 测试要点
- [ ] 商品搜索和筛选
- [ ] 购物车增减商品
- [ ] 订单信息确认
- [ ] 支付方式选择
- [ ] 支付结果回调
- [ ] 订单状态更新

### 依赖数据
- 测试用户账号
- 测试商品（库存充足）
- 测试支付账户
```

### 选择指南

| 格式 | 适用场景 | 优点 | 缺点 |
|-----|---------|-----|-----|
| 简洁型 | 快速执行、smoke测试 | 简洁 | 信息不全 |
| 标准型 | 通用场景 | 完整 | 较冗长 |
| Gherkin | BDD流程、团队协作 | 可读性强 | 编写稍复杂 |
| API型 | 接口测试 | 结构清晰 | 不适合UI测试 |
| 数据驱动 | 参数化测试 | 高效 | 需要工具支持 |
| 场景型 | E2E测试 | 完整 | 复杂度高 |

### 最佳实践

1. **一致性**：整个项目保持统一的格式
2. **可执行性**：每个用例都应能独立执行
3. **明确性**：输入、步骤、预期结果必须清晰
4. **可追溯性**：每个用例应能关联到需求
5. **可维护性**：避免重复，便于更新
